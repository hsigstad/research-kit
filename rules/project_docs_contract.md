# Repository Documentation Contract

This document defines the standard documentation structure used in project and pipeline repositories.

Claude must follow this contract strictly.

---

## 1. Purpose

Each project repository contains a `docs/` directory that serves as the
machine-readable project memory.

The goal is to ensure that:

- documentation looks the same across repos
- updates are deterministic
- commands like “add to meetings” or “add to todos” are unambiguous
- files remain easy to parse, diff, and maintain in git

---

## 2. Repository structure

This section defines the canonical directory layout for research project repositories.

### Canonical structure

```
repo/
  README.md
  docs/
    summary.md
    thinking.md
    todo.md
    done.md
    meetings.md
    feedback.md
    literature.md
    institutions.md
    data.md
    methods.md
    decisions.md
    theory.md        # optional
    hypotheses.md    # optional
    results.md       # optional
    archive.md       # optional
    briefs/          # optional; human-readable topic narratives
    reference/       # optional; structured lookup material (mappings, catalogs)
    notes/           # optional; miscellaneous working documents

  source/          # analysis code
  build/           # all code outputs except paper tables/figures

  paper/           # paper-facing materials (version controlled)
    paper.tex
    appendix.tex
    references.bib
    tables/
    figures/

  talk/            # presentation slides (optional, version controlled)
    talk.tex
    figures/

  references/      # optional; primary source documents (legislation, news, case files)
```

### Semantic role of top-level directories

**docs/**
- Project memory and reasoning artifacts
- Structured research documentation
- Tracked by git

**source/**
- Analysis code only
- Scripts, notebooks, modules
- Substructure (`source/clean/`, `source/assemble/`, `source/figure/`, etc.)
  follows `rules/build_layers.md`
- Tracked by git

**build/**
- All outputs produced by code in `source/`, except paper tables and figures
- Includes: cleaned datasets, intermediate files, model outputs, logs
- Layered into `build/clean/` → `build/assemble/` → `build/analysis/` per
  `rules/build_layers.md`
- Typically gitignored (regenerable from source)
- Must not be treated as authoritative project memory

**paper/**
- Manuscript and all materials referenced in the paper
- Tables and figures used in the paper
- Tracked by git

**talk/** (optional)
- Presentation slides and talk-specific figures
- May reference figures from `paper/figures/` or keep its own in `talk/figures/`
- Tracked by git

**references/** (optional)
- Primary source documents collected during research: legislation, news articles,
  case files, institutional guides, working papers
- Organized by source type (e.g., `news/`, `legislacao/`, `cade/`)
- News articles use a CSV index (`stories.csv`) with full texts saved in a
  `texts/` subdirectory (numbered files), since URLs decay and paywalls appear
- Tracked by git (text files are small; large PDFs may go in Dropbox or be gitignored)
- Not generated by code — these are collected inputs, not outputs

### Output paths for tables and figures

Analysis code should write:
- Tables and figures for the paper directly to `paper/tables/` and `paper/figures/`
- All other outputs to `build/`

This ensures paper-facing outputs are version controlled and co-located with the manuscript, while intermediate and regenerable outputs remain outside version control.

### Behavioral guidance

Claude should:
- Assume this structure when navigating repositories
- Avoid inventing new top-level folders without asking
- Adapt gracefully if an older repo partially deviates from this structure

## 3. Pipeline repository structure

Pipeline repositories are used for data cleaning and processing. They produce
cleaned datasets consumed by multiple projects.

### Canonical structure

```
repo/
  README.md
  docs/
    summary.md
    thinking.md
    todo.md
    data.md
    decisions.md
    archive.md

  source/          # pipeline code
  build/           # outputs (cleaned data, logs, etc.)
```

### Differences from project repositories

- No `paper/` directory (pipelines do not produce papers)
- Fewer `docs/` files (no meetings.md, feedback.md, literature.md, institutions.md, methods.md, results.md)
- `data.md` describes both input and output datasets

### Semantic role of directories

**docs/**
- Pipeline documentation and decision history
- Tracked by git

**source/**
- All pipeline code (scripts, notebooks, modules)
- Tracked by git

**build/**
- Cleaned datasets and processing outputs
- Typically gitignored or written to external storage
- Must be regenerable from source

## 5. Semantic meaning of each file

### summary.md
- Big-picture description of the project
- Research question and motivation
- Conceptual framework
- Design choices and rationale
- High-level structure of the paper or system

This file should change infrequently.

Do not put meeting notes or todos here.

---

### todo.md
Contains actionable tasks only. Only open (uncompleted) tasks belong here.

Used when the user says:
- "add a todo"
- "add this to the todo list"

Organize by topic (`## Topic heading`), not by priority. No prose discussion.

---

### done.md
Contains completed tasks moved from `todo.md`.

When a task is marked as done, it must be moved from `todo.md` to `done.md`. Never leave completed `[x]` items in `todo.md`.

This file serves as a record of what has been accomplished.

---

### meetings.md
Contains factual records of meetings or discussions.

Used when the user says:
- “add to the meetings record”
- “record this meeting”
- “add meeting notes”

No speculation or analysis beyond what was said.

---

### feedback.md
Contains external feedback.

Includes:
- reviewer comments
- seminar feedback
- collaborator comments
- referee reports

Used when the user says:
- “add this feedback”
- “record reviewer comments”
- “log seminar feedback”

---

### literature.md
Contains references relevant to the project.

Includes:
- papers
- talks
- tools
- datasets
- blog posts (if relevant)

Focus on relevance, not citation formatting.

---

### decisions.md
Contains irreversible or important project decisions.

Includes:
- methodological choices
- scope restrictions
- modeling assumptions
- design commitments

Each entry must include:
- decision
- reason
- alternatives considered (if applicable)

---

### thinking.md

This file contains active thinking that has not yet been decided or operationalized.

It includes:
- open questions
- speculative ideas
- alternative designs
- thoughts about how to proceed
- possible connections to literature
- methodological sketches
- doubts and unresolved issues

This file represents **thinking-in-progress**.

Used when the user says:
- “add this to my thinking notes”
- “record this idea”
- “write this down as a thought”
- “add thoughts about how to do X”
- “note a possible connection to literature Y”

Rules:
- Do not treat content in this file as a decision.
- Do not convert content here into todos unless explicitly instructed.
- Do not rewrite or clean aggressively; preserve exploratory language.
- Prefer bullet points over polished prose.

Ideas may later be promoted from `thinking.md` to:
- `todo.md` (if they become actionable)
- `decisions.md` (if they are settled)
- `summary.md` (if they become part of the stable project narrative).

---

### institutions.md

Structured research memory for institutional background.

Contains:
- factual descriptions of legal institutions
- court structure and procedures
- legal processes and rules
- regulatory frameworks
- statutory references

This file is not a paper draft. It is a machine-readable reference document.

Rules:
- Prefer bullet points over prose.
- Keep entries concise, factual, and easy to update.
- Preserve high-signal information, not narrative flow.
- Must not include speculation, interpretation, or empirical results.
- Speculative content belongs in `thinking.md`.

---

### data.md

Structured research memory for data documentation.

Contains:
- data sources and access
- coverage and time period
- unit of observation
- variable definitions and construction
- sample restrictions
- data quality notes

This file is not a paper draft. It is a machine-readable reference document.

Rules:
- Prefer bullet points over prose.
- Keep entries concise, factual, and easy to update.
- Preserve high-signal information, not narrative flow.
- Must not include empirical results or interpretation.
- Speculative content belongs in `thinking.md`.

---

### methods.md

Structured research memory for methodology documentation.

Contains:
- identification strategy
- estimands and parameters of interest
- key assumptions
- robustness strategy
- model specifications

This file is not a paper draft. It is a machine-readable reference document.

Rules:
- Prefer bullet points over prose.
- Keep entries concise, factual, and easy to update.
- Preserve high-signal information, not narrative flow.
- Must not include results or speculation.
- Speculative content belongs in `thinking.md`.

---

### theory.md (optional)

Structured research memory for the formal or theoretical framework.

In theory projects, this file documents the model itself: definitions, assumptions,
key results and propositions, proof strategies, and the mathematical tools being used.

In empirical projects, this file documents the theoretical apparatus that motivates
the empirical design: the structural model (if any), theoretical predictions being
tested, and formal assumptions that come from theory rather than identification.

Contains:
- model setup and definitions
- key assumptions and their justification
- propositions, lemmas, and proof sketches
- mathematical tools and frameworks being imported
- mapping between theoretical objects and empirical counterparts (if applicable)

This file is not a paper draft. It is a machine-readable reference document.

Must not contain:
- speculative or unsettled ideas (those belong in `thinking.md`)
- empirical results or estimation details (those belong in `results.md` / `methods.md`)
- raw derivations or scratch work

Rules:
- Content should only be added when the theoretical component is sufficiently settled.
- Place exploratory or tentative theoretical ideas in `thinking.md` instead.
- Record design choices about the model in `decisions.md`.

---

### hypotheses.md (optional)

Structured mapping of theoretical predictions to evidence and testable implications.

In theory projects, this file maps model predictions to available qualitative or
quantitative evidence and identifies where predictions differ across competing
theories. It organizes the bridge between formal results and empirical reality.

In empirical projects, this file documents the specific hypotheses being tested,
the theories that generate them, the pre-existing evidence for or against each,
and the empirical strategy for adjudication.

Contains:
- hypotheses with the generating theory identified
- predictions: what the theory says should happen
- pre-existing evidence: what is already known (with sources)
- competing predictions: what alternative theories predict differently
- empirical test or evidence strategy
- data requirements and availability
- priority ranking

This file is not a paper draft. It is a machine-readable reference document.

Cross-references `theory.md` for theoretical frameworks, `literature.md` for
sources, and `data.md` for data availability.

Rules:
- Each hypothesis should identify which theory generates the prediction.
- Where possible, identify predictions that discriminate between competing theories.
- Evidence assessments should be honest about strength and limitations.
- Speculative or loosely formed hypotheses belong in `thinking.md` until sharpened.

---

### desiderata.md (optional)

What the project's theory or model should deliver — independent of any specific formalization.

This file articulates the requirements for the theoretical framework: what phenomena
it must explain, what predictions it should generate, what it should enable analysts
to do, and what institutional or policy questions it should illuminate. It serves as
a stable target that persists even if the formal model changes entirely.

Primarily useful for theory projects or empirical projects with a substantial
theoretical component. The document should be framework-agnostic: it describes
*what* the theory must accomplish, not *how* a specific model accomplishes it. It
should not reference existing propositions, notation, or model-specific objects.

Contains:
- phenomena the theory must explain (descriptive adequacy — known facts it must be consistent with)
- testable predictions the theory should generate (empirical bite — new claims that distinguish it from competing theories)
- applications the framework should be useful for (analytical toolkit and institutional design questions it can illuminate)

This file is not a paper draft. It is a requirements document for the theoretical framework.

Cross-references `hypotheses.md` for detailed hypothesis-evidence mapping and
`theory.md` for the settled formal framework.

Rules:
- Content should be stated in terms of what any adequate theory must deliver, not in terms of a specific model's objects or results.
- Do not reference specific propositions, notation, variable names, or file paths from the current formalization.
- Predictions should identify competing theories and what they predict differently.
- Policy/design implications should be framed as questions the model can illuminate, not as specific policy conclusions.
- May be created early in a project to guide model development and updated as understanding sharpens.

---

### outline.md (optional)

Structured plan for the paper's argument and section organization.

This file bridges the gap between structured project memory (`docs/`) and the
finished manuscript (`paper/`). It captures the logical skeleton of the paper:
what each section must establish, what it depends on, and how the overall
argument flows.

Claude should consult this file before drafting or revising prose in `paper/`.
When the outline exists, Claude should flag inconsistencies between the outline
and the manuscript (e.g., a section that assumes a result not yet established).

Contains:
- overall argument structure (claim, evidence, contribution)
- section-by-section plan with purpose and dependencies
- mapping from sections to evidence (tables, figures, results)

This file is not a paper draft. It is a structural plan.

Must not contain:
- finished prose or paragraph drafts
- speculative ideas (those belong in `thinking.md`)
- results or findings (those belong in `results.md`)

Rules:
- Content should reflect settled decisions about paper structure, not brainstorming.
- Exploratory ideas about paper organization belong in `thinking.md` until settled.
- Record major structural decisions (e.g., dropping a section, reordering the argument) in `decisions.md`.
- Update the outline when the paper's scope or structure changes materially.
- The outline may lag behind the manuscript; it is a planning tool, not a mirror.

---

### archive.md (optional)

Stores ideas, thoughts, or directions that are no longer active.

Contains:
- superseded ideas
- abandoned directions
- outdated reasoning
- historical context

This file preserves historical context without affecting current reasoning.

Rules:
- Archive.md must not be treated as active project input.
- Claude should not rely on archived content when reasoning about the current project unless explicitly instructed.
- Content should only be moved into archive.md when explicitly instructed by the user.
- Claude must not archive content automatically.
- Claude must not revive or reuse archived ideas unless the user explicitly requests it.

---

### results.md (optional)

Structured research memory for stabilized empirical findings.

Contains:
- summaries of main results
- null results
- robustness conclusions
- references to tables, figures, or output files (by path)

This file serves as a memory of what has been learned so far.

This file is not a paper draft. It is a machine-readable reference document.

Must not contain:
- raw regression tables or output
- exploratory or intermediate results
- speculative interpretation

Rules:
- Content should only be added to results.md when explicitly instructed by the user.
- Claude must not infer results from output files automatically.
- Speculative or tentative interpretations belong in `thinking.md` instead.
- Superseded or invalidated results should be moved to `archive.md` when explicitly instructed.

---

### docs/briefs/ (optional)

Human-readable narrative documents that tell the cohesive story of a specific topic, analysis, or pipeline.

The canonical `docs/` files are structured memory optimized for targeted lookups and machine consumption. Briefs serve a different purpose: giving a human (or AI) a complete picture of a topic without reading across multiple files.

Contains:
- any `.md` files, freely named (e.g., `linking.md`, `extraction-evaluation.md`)
- no rigid schema or required sections
- narrative prose is acceptable (unlike canonical files which prefer bullet points)

Relationship to canonical files:
- Canonical files remain the authoritative structured memory
- Briefs synthesize and narrate content that may also appear in canonical files
- When facts change, the canonical files are updated first; briefs may lag
- Briefs should not contain information that is absent from all canonical files — if a brief introduces new factual content, that content should also be recorded in the appropriate canonical file

Rules:
- Claude may create briefs when documenting a substantial analysis or pipeline
- Claude should not create briefs for content that fits naturally in a single canonical file
- Briefs do not replace canonical files; they complement them
- No approval needed to add new files under `docs/briefs/` (unlike `docs/` root)

---

### docs/reference/ (optional)

Structured reference material optimized for targeted lookups rather than end-to-end reading.

Unlike briefs (which tell a cohesive story) and canonical files (which are structured memory), reference documents are lookup tables, mappings, catalogs, and checklists that support analysis or coding decisions.

Contains:
- any `.md` files, freely named (e.g., `codebook_mapping.md`, `red_flag_indicators.md`)
- tabular or list-heavy content
- cross-reference mappings between classification systems
- indicator catalogs with testability assessments

Relationship to other docs:
- Reference documents support canonical files and analysis code
- They may be cited from canonical files, briefs, or code comments
- When underlying facts change, the canonical files are updated first; reference docs may need corresponding updates

Rules:
- Claude may create reference documents when producing structured lookup material
- Claude should not create reference documents for narrative content (use briefs) or for structured memory (use canonical files)
- No approval needed to add new files under `docs/reference/` (unlike `docs/` root)

---

### docs/notes/ (optional)

Miscellaneous working documents that do not fit the semantic roles of canonical files, briefs, or reference documents.

This is the catch-all folder for one-off analyses, source-specific notes, exploratory write-ups, reading notes, and any other documentation that needs a home but does not match an existing category.

Contains:
- any `.md` files, freely named
- no rigid schema or required sections
- content may be rough, exploratory, or narrowly scoped

Relationship to other docs:
- Notes are lower-commitment than briefs or reference documents
- Content in notes may later be promoted to a brief, reference doc, or canonical file if it matures
- Notes do not replace canonical files; they supplement them for content that does not fit elsewhere

Rules:
- Claude may create notes freely when no existing file or folder is a natural fit
- No approval needed to add new files under `docs/notes/` (unlike `docs/` root)

## 7. Behavioral rules for Claude

When operating in a project repository:

1. Always check whether `docs/` exists.
2. Use the standardized files above.
3. Never mix content across files.
4. Never invent new documentation files in `docs/` root without asking. Files under `docs/briefs/`, `docs/reference/`, and `docs/notes/` may be created freely.
5. When the user says "add X to Y", update the corresponding file directly.
6. If uncertain where something belongs, ask before writing.

### Rules for todo.md and done.md

7. When marking a task as done, always move it from `todo.md` to `done.md`.
8. Never leave completed `[x]` items in `todo.md`.
9. `todo.md` contains only open tasks; `done.md` contains only completed tasks.

### Rules for institutions.md, data.md, methods.md, and theory.md

10. Treat these files as authoritative structured research memory.
11. Keep content factual and concise; avoid narrative prose.
12. Place speculative or tentative content in `thinking.md` instead.
13. Record decisions derived from these files in `decisions.md`.

### Rules for archive.md

14. Only move content to archive.md when explicitly instructed by the user.
15. Never archive content automatically.
16. Never revive or reuse archived ideas unless the user explicitly requests it.
17. Do not rely on archived content when reasoning about the current project.

### Rules for results.md

18. Only add content to results.md when explicitly instructed by the user.
19. Never infer results from output files automatically.
20. Place speculative or tentative interpretations in `thinking.md` instead.
21. Move superseded or invalidated results to `archive.md` only when explicitly instructed.
